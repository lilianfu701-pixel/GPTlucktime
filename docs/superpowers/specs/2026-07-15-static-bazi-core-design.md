# 静态八字命盘核心层设计

## 1. 目标与边界

本阶段交付一个可复现、可追溯的四柱八字静态出生盘计算内核。给定同一份出生资料与同一算法/数据版本，结果必须保持不变。

本阶段只计算出生瞬间固定的事实，不输出吉凶、断语、建议、概率或推荐。

纳入范围：

- 公历出生日期与当地民用时间。
- 出生地经纬度、IANA 时区、历史 UTC 偏移和夏令时规则。
- 真太阳时：当地民用时间、标准时差、经度校正、地方平太阳时与均时差。
- 儒略日、精确节气时刻、立春与节的边界判定。
- 年、月、日、时四柱；干支索引、纳音、旬空、十二长生。
- 藏干、十神、五行统计、月令、通根与结构关系。
- 天干合冲与地支六合、六冲、刑、害、破、三合、三会、半合、墓库。
- 基础神煞仅作为规则命中结果输出，不附带吉凶语义。
- 输入规范化、规则版本、数据来源版本与逐步计算追踪。

明确不纳入范围：

- 大运、流年、流月、流日、流时与任意动态时间推演。
- 生活地对运势的影响；生活地仅预留为未来动态层的上下文。
- 格局裁定、用神/喜忌结论、自动解读、断语、投资或博彩建议。
- 紫微斗数、奇门、六爻、太乙等其他体系。

## 2. 技术方案

采用 TypeScript 的纯函数领域内核，Node.js 测试环境使用 Vitest。第一版不依赖数据库、网络请求或界面；所有外部历法与时区资料在适配层封装，核心规则只依赖明确类型的输入。

方案选择：

1. 纯 TypeScript 内核加可替换的历法/天文适配器（采用）。可精确测试、可锁定版本、以后可接 API 或界面。
2. 直接使用单一命理库。启动较快，但关键边界难审计，规则口径受第三方限制。
3. 先做前端表单。可演示但会掩盖历法和规则正确性，后续返工风险高。

## 3. 架构与数据流

```text
BirthInput
  -> normalizeBirthInput
  -> resolveCivilTime (IANA 历史时区、DST、歧义处理)
  -> calculateSolarTime (经度校正、均时差、真太阳时)
  -> calculateCalendarContext (JDN、精确节气、边界)
  -> calculatePillars (年/月/日/时柱)
  -> deriveStaticIndicators (藏干、十神、五行、关系、辅助指标)
  -> ChartContext { chart, trace, warnings, versions }
```

模块边界：

| 模块 | 责任 | 不负责 |
| --- | --- | --- |
| `core/types` | 输入、输出、错误和追踪类型 | 具体算法 |
| `engines/time` | 民用时间、时区、夏令时与真太阳时 | 节气、干支 |
| `engines/calendar` | JDN、太阳黄经、节气边界 | 命理解释 |
| `engines/pillars` | 四柱与六十甲子 | 五行强弱结论 |
| `engines/indicators` | 藏干、十神、五行、关系、辅助表 | 吉凶断语 |
| `chart/build-chart-context` | 编排与不可变输出 | 规则细节 |

## 4. 输入、输出与时间口径

### BirthInput

必填字段：

- `localDateTime`: 出生地民用日期与时间，精度可到秒。
- `timeZone`: IANA 时区，例如 `Asia/Shanghai`。
- `birthplace`: 名称、纬度与经度。

可选字段：

- `gender`：只存储；静态盘计算不使用。
- `timePrecision`：`exact | approximate | unknown`。
- `civilTimeResolution`：面对 DST 重复时刻时，要求明确为较早或较晚偏移；面对不存在时刻时拒绝输入。
- `residenceContext`：只记录，不进入静态盘计算。

时间规则：

1. 先将民用时间按 IANA 历史规则解析为 UTC instant。
2. DST 跳过的当地时刻必须返回可读错误；DST 重复的当地时刻必须由调用方消歧，否则返回可读错误。
3. 地方平太阳时修正为 `(longitude - standardMeridianLongitude) * 4 minutes`。
4. 真太阳时为地方平太阳时加均时差；均时差和节气均基于同一 UTC instant。
5. 日柱与时柱的换日口径第一版固定为“真太阳时 00:00 换日”。子初换日、子正换日作为 P1 可配置规则，不能混入默认结果。
6. 年柱以立春瞬时为界；月柱以十二节瞬时为界，不以农历月份替代。

### ChartContext

输出必须包含：

- 标准化输入与 UTC instant。
- 解析出的 UTC 偏移、DST 偏移、标准子午线、经度修正、均时差、地方平太阳时和真太阳时。
- JDN、前后节气、当前节气、每个相关边界的距离和风险提示。
- 四柱、六十甲子索引、藏干、十神、五行计数和规则命中关系。
- 纳音、旬空、十二长生和基础神煞命中项。
- `trace`：每个派生指标的公式/规则名、输入、结果和版本。
- `versions`：应用、规则表、天文算法、时区数据库和节气数据版本。
- `warnings`：未知出生时刻、节气/时辰/换日边界邻近、手工覆盖等不确定性。

## 5. 准确性与错误处理

“准确”在本项目中是可核查的计算准确性，而不是对人生结果的保证。具体约束如下：

- 时区数据使用 IANA 标识和固定版本；不接受仅写“东八区”而未确认历史规则的输入。
- 天文适配器对外声明算法与有效日期范围；范围外拒绝计算或明确降级，绝不静默伪精确。
- 节气按 UTC instant 比较，不以格式化后的日期/时区字符串比较。
- 每个表驱动规则均有唯一规则 ID、来源说明和版本号。
- 输入不完整、坐标越界、无效 IANA 时区、DST 歧义和不可解析日期必须返回结构化错误。
- 计算不保留可变全局状态，调用顺序不得改变结果。

## 6. 规则清单

P0 的规则输出为事实项，不含解释文本：

| 规则组 | P0 输出 |
| --- | --- |
| 四柱 | 年/月/日/时的天干、地支、60 甲子索引 |
| 藏干 | 每个地支的主气/中气/余气与权重来源 |
| 十神 | 相对日主的五行生克和阴阳关系 |
| 五行 | 显性干支、藏干的元素计数；不作强弱吉凶判断 |
| 月令/通根 | 月支季节信息、天干在各支的根状态 |
| 合冲刑害等 | 参与柱位、命中规则 ID 与关系类型 |
| 辅助表 | 纳音、旬空、十二长生、经审定的基础神煞 |

日主强弱、得令、得地、得助、扶抑和调候会保留原始特征供 P1 规则层使用；P0 不输出“身强/身弱”之类的裁定。

## 7. 测试策略与验收标准

开发采用测试先行。测试集按以下层级建立：

- 单元测试：干支循环、藏干、十神、关系规则、经度修正、均时差接口、输入验证。
- 集成测试：出生输入到 `ChartContext` 的完整纯计算链。
- 黄金样例：已核对的普通日期、节气瞬间前后、真太阳时跨时辰/跨日、历史 DST、DST 重复与跳过时刻。
- 性质测试：60 甲子循环、时间与空间边界不变量、重复调用一致性。

验收条件：

1. 每个 P0 指标都能在输出 `trace` 中找到规则和输入来源。
2. DST 无效/歧义输入不产生静默结果。
3. 节气边界前后样例切换正确，且警告中包含边界距离。
4. 相同输入与版本在重复运行中得到字节级等价的规范化输出。
5. 单元加集成测试的语句/分支覆盖率均不低于 80%，没有跳过测试。

## 8. 分阶段交付

第一开发批次建立工程骨架、领域类型、数据表、干支与十神纯函数，以及对应 RED/GREEN 测试。第二批次接入时间与节气适配器并完成四柱。第三批次完成静态指标、追踪和黄金样例。每批次独立验证，不跨越动态推演范围。
